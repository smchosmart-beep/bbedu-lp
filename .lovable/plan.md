# 보정판 적용 시 위험 재검토

코드 실측(L1084, L1776, L1822~1840, L1889, L1957, L2119) 기준으로 항목별로 (1) 기능 오작동, (2) 서버비 역증, (3) 다른 기능 영향 세 축으로 점검.

---

## 1차 (즉시 적용) — 모두 안전

### 1-D. `RAG_KEEP_RECENT 2 → 1`
- 오작동: 없음. 압축된 메시지도 tool role/이름 유지 → tool_call id 매칭 깨지지 않음(이미 현행 코드가 동일 패턴으로 압축).
- 비용 역증: 없음. 캐시(`_b3.ragCache`)는 stage가 같으면 보존(L1826 회전은 stage 바뀔 때만) → 같은 단계 재참조는 무료 hit.
- 다른 기능: present_choices·hwpx·미리보기 무관(상태는 `partialPlan`/`confirmedChoices`).

### 1-E. RAG 응답 트리밍
- 오작동: `find_standards.해설` 80자 컷은 폴백 경로(L1906)에 이미 적용 중인 동일 규칙. 사용자 안내문은 `showStandardGuidance()`(L2190)가 `standard_guidance.json` 원문 사용 → 사용자 노출은 영향 0.
- `list_lesson_models.설명` 제거는 **단계 있는 모형만**. 단계 없는 모형은 설명이 유일 식별자라 유지 → 활동 자동 작성(이전 턴 수정 사항) 영향 없음.
- 비용 역증/타 기능 영향: 없음.

### 3-B. `complete_plan` 사전 차단
- 오작동: 정규식이 `placeholder|예시|TODO|^\s*\(\s*\)\s*$`로 좁음. `\.{3,}`(말줄임) 제외해 합법 본문 오탐 없음. 차단 시 LLM에 필드명+사유 반환 → 1회 update_plan으로 회복.
- 비용 역증: 없음. 오히려 verify 2콜(A+B) → 0콜.
- 다른 기능: `verifyPlanQuality`가 호출 안 돼도 placeholder 통과 위험 없음(가드 자체가 동일 검출).

### 3-C. stage 10 단일 update_plan
- 오작동: 도구 스키마·핸들러 무변경(프롬프트 텍스트만 추가). 분할 호출이 들어와도 동작은 유지(권장 문구일 뿐).
- 비용 역증: 없음. 평균 3→1 콜.
- 다른 기능: 자동 작성 누락 위험 → tool_choice가 stage 10에서 `auto`(이전 적용)라 모델이 자체 판단으로 한 번에 작성 가능. **모니터**: 누락 사례가 보이면 "한 응답에서 1회만"을 "권장"으로 약화.

### 2-B. 채팅 3문장 캡
- 오작동: 도구 호출 흐름 무관. 도구 미포함 일반 안내만 짧아짐.
- 비용 역증: 없음.
- 다른 기능: standard guidance·확인 멘트는 별도 함수(클라)라 영향 없음.

### 3-D 대체 (캐시 hit는 stage 6 카운트에서 제외)
- 오작동: L1832 한 줄(카운트++ 조건부). hit 시 stop 가드도 그대로 동작.
- 비용 역증: 없음. 실제 신규 호출만 한도 소모 → 정상 진행이 stop hint에 잘리는 사고 ↓.
- 다른 기능: backward 설계 ⓪~③ 중단 위험 해소.

**1차 결론**: 그대로 한 묶음 적용 가능. 회귀 신호 발생 시 각 스위치 1줄 OFF로 복귀.

## 2차 (24h 후) — 1-C 대체에만 추가 가드 필요

### 1-A 보정 (`stage-1` 제거만)
- 오작동: stage 전이 직후 한 턴은 partialPlan으로 직전 결과 복구 가능(필드 기반). 다만 **stage 6 진입 직후 첫 턴**에서 stage 5(요소 선택) 메시지가 사라지면 평가 요소 후보를 다시 묻는 회귀 가능성 → confirmedChoices가 stage 5 선택을 보유하므로 실제 재질문 차단됨(L1776 가드).
- 비용 역증: 없음.
- 다른 기능: 영향 없음.

### 1-B 보정 (CORE 2분할, stage≥8만 LATE 주입)
- 오작동: `[필드 작성 규칙]`(◉◦-, sub키, 시간/차시) 미주입 시 stage 1~7에서 ◉/◦ 마크 오용 → 미리보기 렌더에 영향. **확인 필요**: 1~7단계에서 LLM이 실제로 ◉/◦ 기호를 쓰는 필드(예: 학습목표·평가 요소 입력 보조)가 있는지. 있으면 `CORE_ALWAYS`에 마크 규칙 1줄(50자)만 잔류.
- 비용 역증: 없음.
- 다른 기능: stage 8~10은 현행과 동일.

### 1-C 대체 (평가 필드 1줄 압축)
- 오작동 위험: **여기만 주의**. `buildAPIMessages`에서 평가 필드를 1줄 압축해 system 메시지로 주입할 때, 원본 `partialPlan`은 그대로 둬야 함(present_choices 가드·hwpx·미리보기는 원본 참조). 압축은 LLM 인풋용 뷰에만 적용.
- 추가로, 압축 포맷이 confirmedChoices 비교에 사용되지 않도록 필드명 그대로 유지(`평가{i}_요소`, `_방법` 등 sub키 보존). 그렇지 않으면 "이미 정한 거 또 물어봄" 회귀.
- 비용 역증: 없음.
- 안전 가드 추가: 압축 후 길이가 원본보다 길면 원본 사용(가드 1줄). 회귀 시 `EVAL_FIELDS_COMPACT=false`.

**2차 결론**: 1-A·1-B는 안전. 1-C 대체는 "압축은 인풋 뷰만, 원본 상태 불변, sub키명 보존, 길이 역전 시 원본" 4가지 가드 포함 시 안전.
