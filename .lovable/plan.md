## 문제 진단

**1) 채팅에 `fields: [{"key":"...","value":"..."}]` 원문이 표시되는 현상**
- `app35.js`의 `addBot()` (327~338줄)은 모델이 채팅에 적은 `[도구 결과: …]` 패턴만 제거하고 있음.
- 모델이 `update_plan` 호출과 별도로 "반영합니다"라는 멘트 끝에 인자 일부(`fields: [...]`, `{"fields":...}`, 또는 raw `[{"key":"…","value":"…"}]`)를 그대로 적어버리는 경우는 거르지 못함.
- 시스템 프롬프트에 "도구 인자/결과를 본문에 적지 말라"는 지시가 있지만 모델이 종종 어김 → 클라이언트 측에서 표시 단계에 한 번 더 청소해야 함.

**2) 활동 시간 합이 40분을 초과해 최종에 남는 현상**
- `doCompletePlan()` 시간 게이트(1930~1938줄)가 **38~42분** 범위까지 통과시킴. 41·42분도 "OK"로 확정되어 그대로 다운로드됨.
- 또한 `update_plan`이 호출되는 매 턴마다 시간 합을 검증하지 않음 → 모델이 잘못된 시간으로 채워두고 11단계로 넘어가도 사용자는 미리보기에서 "총 45분"인 상태를 한참 보게 됨(샘플 메시지의 상황).
- 모델 자율 보정에 의존하다 보니 "10분으로 줄여서 반영하겠습니다"라고 말만 하고 실제 `update_plan` 호출은 빠지거나 잘못된 sub만 줄이기도 함.

## 적용할 변경

### A. 채팅 누출 정리 강화 — `public/legacy/app35.js` `addBot()`
기존 `[도구 결과: …]` 제거 정규식 다음에 아래 정리 단계 추가:

1. `^\s*fields\s*:\s*[\[\{].*$` — `fields: [...]` 또는 `fields: {...}`로 시작하는 줄 전체 제거 (다중행 JSON은 같은 줄에 한 덩어리로 오는 케이스가 대부분).
2. `^\s*\{?\s*"fields"\s*:[\s\S]*?\]\s*\}?\s*$` — `{"fields":[…]}` 한 줄 블록 제거.
3. `^\s*\[?\s*\{\s*"key"\s*:[\s\S]*?\}\s*\]?\s*$` — 키/값 페어가 단독 줄로 새는 패턴 제거.
4. `^\s*\{?\s*"ok"\s*:\s*(true|false)[\s\S]*?\}\s*$` — `{"ok":true}` 같은 도구 결과 에코 제거.
5. 연속 공백 줄 정리(`\n{3,}` → `\n\n`)는 그대로 유지.

규모는 1줄 → 5~6줄 추가. 다른 본문 텍스트(예: 사용자가 적은 평문)는 영향 없음 — 모두 줄 단위 `^…$` 매칭.

### B. 시간 합 게이트 강화 — `doCompletePlan()` (1930~1938줄)
- 허용 범위 38~42 → **정확히 40분** 으로 강화. (실수업이 40분 단위라 사용자가 명시한 규칙)
- 메시지에 "마지막 활동(`전개_sub{N}_시간`)에서 초과분만큼 줄여 update_plan 후 다시 호출" 가이드를 명시해 모델이 어느 필드를 고쳐야 하는지 모호함을 제거.

### C. 매 턴 시간 합 사전 점검(소프트) — 모델에 즉시 피드백
`runTool('update_plan', …)`이 시간 관련 키(`*_시간`)를 건드린 직후, 합을 계산해 40을 넘으면 `update_plan` 결과의 `ok:true`에 `warn` 필드를 덧붙여 반환(차단은 아님):
```js
{ ok: true, warn: "현재 시간 합 45분(40 초과). 가장 긴 전개_sub 시간을 줄여 40에 맞추세요." }
```
- 차단(`ok:false`) 대신 경고로 두는 이유: 모델이 도입·전개를 분할 호출하는 도중에는 일시적으로 합이 안 맞을 수 있음. 11단계 진행 자체를 막으면 회귀(재시도 비용 증가) 위험.
- 모델은 자기 도구 결과의 `warn`을 읽고 다음 호출에서 보정 → "45분으로 한참 보이는" 현상을 빠르게 줄임.

### D. 시스템 프롬프트 보강(작게)
- 10단계 지시문(161줄) 끝에 한 줄 추가:
  - "**시간 합은 정확히 40분(±0). 합이 다르면 가장 긴 `전개_sub{i}_시간`만 보정해서 `update_plan`을 다시 호출하세요. 채팅에 `fields:` JSON을 절대 적지 마세요(도구 호출만으로 표현).**"

## 영향 검토

- **A (정규식 청소)**: 표시 전용. 히스토리(`state.messages`)는 원문 그대로 유지되므로 모델 맥락 손실 없음. 사용자가 직접 입력한 일반 텍스트는 `^fields:`/`^{"fields"…}` 같은 단독 줄 패턴과 겹칠 일이 사실상 없음.
- **B (정확 40분)**: 검수 단계에서만 동작. 통과 못 하면 모델이 자동 보정 → 1회 재시도 비용은 들지만, 잘못된 시간을 가진 최종 산출물을 막는 것이 우선. 기존 사용자가 이미 40분 표준을 지키려 하던 흐름이므로 추가 재시도 빈도는 낮을 것.
- **C (warn)**: 차단이 아니라서 비용·진행에 부담 없음. 모델이 다음 턴에 자율 보정할 단서만 제공.
- **D**: 프롬프트 한 줄 추가 → 입력 토큰 미미.
- **다른 단계(검수·검토 99·100, RAG, 비용 로깅, retryByStage)**: 손대지 않음. 직전 비용 최적화 변경과 충돌 없음.
