# 2차 부분 적용 (1-A + 1-B) 적용 시 위험 재검토

코드 실측(L106~115 시스템 블록, L1085~1107 buildAPIMessages, L1393~1566 messages.push 사이트들) 기준으로 두 항목의 (1) 기능 오작동, (2) 서버비 역증, (3) 다른 기능 영향 점검.

---

## 1-A. 오래된 stage 메시지 드롭

### ⚠ 구현상의 핵심 리스크
`state.messages` 에는 stage 태그가 없음(push 사이트 10여 곳 모두 `{ role, content }` 또는 `{ role, tool_calls }` 만). "stage ≤ 현재-3 메시지를 드롭" 하려면 **모든 push 사이트에 stage 태깅을 먼저 추가**해야 함. 한 곳이라도 누락되면 그 메시지는 stage=undefined 로 영구 잔류하거나(보수 해석) 항상 드롭(공격 해석)됨 — 양쪽 다 회귀.

→ **권고**: 1-A 는 "stage 태깅 작업 + 윈도잉 적용"의 2-PR 로 분리. 태깅만 먼저 적용해 1~2일 모니터(누락된 사이트 자동 감지: stage 누락된 user/assistant 메시지 카운트 로깅) 후에 드롭 활성화.

### orphan tool_call 위험 (API 400 → 재시도 → 비용 ↑)
OpenAI 호환 API는 `assistant.tool_calls` 와 `tool` 메시지의 1:1 매칭을 요구. stage 컷오프가 assistant(tool_calls) 와 그 tool 응답 사이를 가르면 API 400. 적응 함수 `adaptMessages`(bridge L138~)가 tool→text 로 평탄화하므로 서버 단에서는 살아남지만, 클라이언트가 직접 OpenAI 스키마로 보내는 경로(/api/lessonplan/chat 의 raw passthrough 여부) 확인 필요.

→ **가드 필수**: drop 단위는 "assistant + 그에 매달린 모든 tool" 한 묶음. 묶음 단위로 keep/drop 결정.

### partialPlan 재주입 덕에 대부분 안전
L1099~1106 이 매 콜 `filled` 를 system 메시지로 재주입 → 확정 필드 손실 없음. 단 **확정 안 된 진행 중 정보**(예: 사용자가 방금 던진 맥락·교사가 강조한 표현)는 history 에서만 살아있음. stage 5→6 전이 직후 한 턴은 stage 5 의 평가요소 후보 토론 컨텍스트가 사라질 수 있음 — L1776 의 confirmedChoices 가드가 재질문은 차단하지만, "왜 그 평가요소를 골랐는지" 같은 문맥은 휘발.

### 비용 역증 시나리오
- orphan tool_call → API 400 → tier 격상 재시도 → **콜당 비용 PRIMARY 등급**.
- 평균 콜은 줄지만 회귀 1건이 평소 5콜분 비용 잡아먹음.

## 1-B. `[필드 작성 규칙]` 블록 stage 8+ 한정

### ⚠ L115 누락이 stage 1 환각 유발
L115 "차시·교과서쪽수·대상학급·일시는 비워 둡니다(임의 생성하지 않음)" — 이 한 줄은 **stage 1 에 결정적**. admin/35 에서 확인된 회귀 패턴(차시 번호·교과서 쪽수 환각)을 막는 유일한 가드. 통째로 stage 8 까지 미루면 회귀.

→ **필수**: L115 한 줄(+ L110 시간 키 규칙 중 통합 키 부분)은 `CORE_ALWAYS` 잔류. ◉/◦ 위계 기호·sub키·(자)(유)(평) 형식만 stage ≥ 8 로 미룸.

### ◉/◦ 사용처 점검 결과
- Stage 6 평가요소(L145): "쉼표·줄바꿈 구분" — ◉/◦ 미사용. 안전.
- Stage 5·7 학습목표·탐구질문: 평문. 안전.
- Stage 1~7 어디서도 ◉/◦ 글머리를 LLM 이 쓸 일 없음. → ◉/◦ 규칙(L107~108, 113~114) 지연은 안전.

### 비용 역증
없음. 시스템 프롬프트만 짧아짐.

### 다른 기능 영향
- 미리보기 렌더(L2025~2030, L2445, L2580)는 LLM 출력에서 ◉/◦ 정규화 — stage 1~7 출력에 ◉/◦ 가 없으면 변환 없음 = 영향 없음.
- HWPX 변환은 stage 10 이후라 무관.

## 종합

| 항목 | 오작동 | 비용 역증 | 타 기능 영향 |
|---|---|---|---|
| 1-A 윈도잉 | **중**(stage 태깅 누락·orphan tool_call) | **저~중**(회귀 시 PRIMARY 재시도) | 저 |
| 1-B 블록 분할 | **저**(L115 잔류 시 무해) | 없음 | 없음 |

### 보정 권고
1. **1-B 만 먼저 적용** — L115·시간 통합키 규칙은 CORE_ALWAYS 잔류, ◉/◦·sub키·(자)(유)(평)·"[활동 N]" 헤더 규칙만 stage ≥ 8 로 미룸. 회귀 위험 ≈ 0, 추가 5~10% 감축 기대.
2. **1-A 는 2-PR 분리**:
   - PR-A1: 모든 `state.messages.push` 사이트에 stage 태깅 + 누락 감지 로깅 추가, 드롭 비활성. 24h 관찰.
   - PR-A2: 누락 0 확인 후 윈도잉 활성화. 드롭 단위는 "assistant + 매달린 tool 묶음". 압축 후 길이 < 원본 60% 면 원본 사용. stage 5→6 첫 진입 턴은 보류.

### 서버비 역증 시나리오
- 1-B 단독: 없음.
- 1-A 활성화 후 orphan tool_call 1회당 ~5콜분 손실. 태깅 검증 없이 바로 적용 시 누적 비용 역전 가능.

### 현재 구현 영향
- 1-D(L1084), 3-D 대체(L1813)와 동선 정합 — 1-B 는 동일 파일 다른 블록 수정이라 충돌 없음.
- 1-A 는 messages 구조를 바꾸므로 admin/35 통계(`stage` 필드)와 정합 필요 — 태깅 시 같은 stage 값 사용.

승인하시면 **1-B 만** 1차 보강으로 적용. 1-A 는 별도 승인 후 2-PR 진행.
